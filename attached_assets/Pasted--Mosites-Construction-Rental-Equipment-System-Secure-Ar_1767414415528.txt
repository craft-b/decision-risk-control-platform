## Technology Stack

### Frontend
- **React 18** with TypeScript
- **Tailwind CSS** for styling
- **React Router v6** for navigation
- **React Query** (TanStack Query) for server state
- **Zustand** for client state (auth, UI state)
- **React Hook Form** + Zod for form validation

### Backend
- **Node.js** with **Express.js**
- **TypeScript**
- **Prisma ORM** (excellent SQL Server support + prevents SQL injection)
- **SQL Server Express** (existing infrastructure)
- **JWT** for authentication
- **bcrypt** for password hashing (cost factor: 12)
- **helmet** for security headers
- **express-rate-limit** for DoS protection
- **express-validator** for input sanitization

### Architecture Pattern
- **RESTful API**
- **Layered Architecture**: Controller → Service → Repository
- **Role-Based Access Control** (RBAC)
- **Defense in Depth** security model

## Security Architecture Overview

### Security Layers
1. **Network Layer**: HTTPS only, CORS restrictions
2. **Application Layer**: Rate limiting, security headers
3. **Authentication Layer**: JWT with refresh tokens, password policies
4. **Authorization Layer**: Role-based + resource-level permissions
5. **Data Layer**: Parameterized queries (Prisma), encryption at rest
6. **Audit Layer**: Comprehensive logging of security events

### Security Principles Applied
- **Principle of Least Privilege**: Users get minimum necessary permissions
- **Defense in Depth**: Multiple security layers
- **Fail Securely**: Errors don't expose sensitive information
- **Secure by Default**: All endpoints require authentication unless explicitly public
- **Input Validation**: Client-side AND server-side validation
- **Output Encoding**: Prevent XSS attacks
- **Audit Logging**: Track all sensitive operations

## Comprehensive Security Implementation

### 1. Authentication Security

#### Password Security
```
- Minimum 12 characters (configurable)
- Require: uppercase, lowercase, number, special character
- bcrypt with cost factor 12 (or argon2 for enhanced security)
- Password history: prevent reuse of last 5 passwords
- Account lockout: 5 failed attempts → 15 min lockout
- Password reset tokens: expire in 1 hour, single-use
```

#### JWT Token Strategy
```
Access Token:
- Short-lived (15 minutes)
- Stored in memory only (React state)
- Contains: userId, role, permissions
- Signed with HS256 (or RS256 for enhanced security)

Refresh Token:
- Long-lived (7 days)
- Stored in httpOnly, secure, sameSite cookie
- Rotation on each use (prevents token replay)
- Stored in database with user association
- Can be revoked by admin

Implementation:
- Access token in Authorization header
- Automatic refresh before expiration
- Silent refresh mechanism
- Token revocation list for logged-out tokens
```

#### Session Management
```
- Session timeout: 30 minutes of inactivity
- Warning at 25 minutes
- Forced logout on timeout
- Single session per user (optional, configurable)
- Session tracking in database
```

### 2. Authorization Security

#### Role-Based Access Control (RBAC)
```typescript
// Roles
enum Role {
  ADMINISTRATOR = 'ADMINISTRATOR',
  VIEWER = 'VIEWER'
}

// Permissions (granular)
enum Permission {
  // Equipment
  EQUIPMENT_READ = 'equipment:read',
  EQUIPMENT_CREATE = 'equipment:create',
  EQUIPMENT_UPDATE = 'equipment:update',
  EQUIPMENT_DELETE = 'equipment:delete',
  
  // Rentals
  RENTAL_READ = 'rental:read',
  RENTAL_CREATE = 'rental:create',
  RENTAL_UPDATE = 'rental:update',
  RENTAL_DELETE = 'rental:delete',
  
  // Swaps
  SWAP_CREATE = 'swap:create',
  SWAP_READ = 'swap:read',
  
  // Reports
  REPORT_VIEW = 'report:view',
  REPORT_EXPORT = 'report:export',
  
  // Admin
  USER_MANAGE = 'user:manage',
  AUDIT_VIEW = 'audit:view'
}

// Role to Permission mapping
const RolePermissions = {
  ADMINISTRATOR: [/* all permissions */],
  VIEWER: [
    Permission.EQUIPMENT_READ,
    Permission.RENTAL_READ,
    Permission.SWAP_READ,
    Permission.REPORT_VIEW
  ]
}
```

#### Resource-Level Authorization
```
- Users can only access resources they own or are assigned to
- Middleware checks: userOwnsResource() or hasPermission()
- Example: Viewer can't edit ANY rental, even if they can see it
```

### 3. Input Validation & Sanitization

#### Server-Side (Critical - Never Trust Client)
```
Every endpoint validates:
1. Data type validation (express-validator)
2. Format validation (regex for emails, phone, etc.)
3. Range validation (dates, numbers)
4. Length validation (prevent buffer overflow)
5. Whitelist validation (enum values)
6. SQL injection prevention (Prisma handles this)
7. XSS prevention (sanitize HTML)
8. Path traversal prevention
9. Command injection prevention

Example validations:
- Asset ID: alphanumeric, max 50 chars
- Dates: valid ISO format, logical range
- Amounts: positive numbers, max 10 digits
- Text fields: max length, no script tags
```

#### Client-Side (User Experience)
```
- Zod schemas for forms
- Real-time validation feedback
- Sanitize display of user-generated content
- Use DOMPurify for any HTML rendering
```

### 4. API Security

#### Rate Limiting
```
Global:
- 100 requests per 15 minutes per IP

Endpoint-specific:
- Login: 5 attempts per 15 minutes per IP
- Password reset: 3 requests per hour per email
- Report generation: 20 per hour per user
- File uploads: 10 per hour per user

Implementation:
- express-rate-limit with Redis store
- Different limits for authenticated users
- Bypass for internal services
```

#### CORS Configuration
```javascript
const corsOptions = {
  origin: process.env.ALLOWED_ORIGINS.split(','), // Whitelist only
  credentials: true, // Allow cookies
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-CSRF-Token'],
  maxAge: 86400 // 24 hours
}
```

#### Security Headers (Helmet.js)
```javascript
helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'unsafe-inline'"], // Minimize unsafe-inline
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", "data:", "https:"],
      connectSrc: ["'self'", process.env.API_URL],
      fontSrc: ["'self'"],
      objectSrc: ["'none'"],
      mediaSrc: ["'self'"],
      frameSrc: ["'none'"],
    },
  },
  hsts: {
    maxAge: 31536000, // 1 year
    includeSubDomains: true,
    preload: true
  },
  noSniff: true, // X-Content-Type-Options
  xssFilter: true, // X-XSS-Protection
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' }
})
```

#### CSRF Protection
```
- CSRF tokens for all state-changing operations
- Double-submit cookie pattern
- Validate token on POST/PUT/DELETE/PATCH
- Token rotation per session
```

### 5. Data Security

#### Encryption
```
At Rest:
- SQL Server Transparent Data Encryption (TDE)
- Encrypt sensitive fields: SSN, payment info (if any)
- Use AES-256-GCM

In Transit:
- HTTPS/TLS 1.3 only
- Strong cipher suites only
- HSTS enabled

In Memory:
- Clear sensitive data after use
- No logging of passwords/tokens
```

#### Database Security
```
- Principle of least privilege for DB user
- Separate users for read vs write operations
- Connection pooling with limits
- Prepared statements only (Prisma handles this)
- No dynamic query building with user input
- Regular backups with encryption
- Backup access controls
```

#### Application Logging
```
Winston logger with levels:
- error: System errors, security events
- warn: Deprecations, potential issues
- info: General information
- debug: Development only

Do NOT log:
- Passwords (plain or hashed)
- Tokens
- Session IDs
- Credit card numbers
- PII beyond what's necessary
- Full stack traces in production
```

#### Monitoring & Alerts
```
Monitor for:
- Repeated failed login attempts
- Unusual access patterns
- Rate limit violations
- Database errors
- High error rates
- Slow queries
- Disk space
- Memory usage

Alert on:
- Security events (failed auth, authorization)
- System errors
- Performance degradation
```

### 7. Error Handling

#### Secure Error Responses
```
Production errors return:
- Generic message: "An error occurred"
- Error code: "ERR_500_INTERNAL"
- Request ID for support

Do NOT expose:
- Stack traces
- Database errors
- File paths
- Configuration details
- Library versions
- Internal structure

Development errors can be verbose
```

#### Package Management
```

- Use exact versions (no ^ or ~)
- Review dependencies before adding
- Use Snyk or Dependabot for alerts
```

## API Security Examples

### Protected Route Pattern
```typescript
// All modification routes follow this pattern:
router.post('/equipment',
  authenticate,                    // JWT validation
  requireRole(Role.ADMINISTRATOR), // Role check
  rateLimit({ max: 50 }),         // Rate limiting
  validateRequest([               // Input validation
    body('assetId').isAlphanumeric().isLength({ max: 50 }),
    body('category').isString().trim().escape(),
    // ... more validations
  ]),
  sanitizeInputs,                 // XSS prevention
  auditLog('CREATE_EQUIPMENT'),   // Audit logging
  equipmentController.create
);
```

### Input Validation Example
```typescript
// Equipment creation validation
const equipmentValidation = [
  body('assetId')
    .trim()
    .isLength({ min: 1, max: 50 })
    .matches(/^[A-Z0-9-]+$/)
    .withMessage('Asset ID must be alphanumeric with hyphens'),
  
  body('category')
    .trim()
    .isIn(['Heavy Equipment', 'Tools', 'Vehicles', 'Other'])
    .withMessage('Invalid category'),
  
  body('serialNumber')
    .optional()
    .trim()
    .isLength({ max: 100 })
    .matches(/^[A-Z0-9-]+$/),
  
  body('purchasePrice')
    .optional()
    .isFloat({ min: 0, max: 10000000 })
    .withMessage('Invalid purchase price'),
];
```
